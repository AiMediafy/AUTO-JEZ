<h2>News: WebSockets in Responses API – Agent Workflow Gains Speed</h2><p>OpenAI has just demonstrated how <strong>WebSockets</strong> in the Responses API can significantly improve performance in agent-driven processes (Codex agent loop). Their technical summary is not just theory – from my perspective, it’s one of the more practical improvements for developers and companies using AI for task automation.</p><h2>How does it work in practice?</h2><ul><li><strong>WebSocket instead of REST</strong>: Traditional APIs (REST) require the client to continuously poll the server for a response. WebSockets allow for maintaining a constant, bidirectional connection – responses arrive instantly, without polling. The key here is less network load and lower latency.</li><li><strong>Connection-scoped caching</strong>: Data/conversation context is cached for the duration of the connection. In practice, this means the model doesn’t have to process everything from scratch at every stage – boosting efficiency and reducing redundant prompt loading.</li><li><strong>Reduced API overhead</strong>: Eliminating unnecessary queries and caching reduces the number of requests by up to half. This translates into real cost and time reductions (in selected use cases: 15–30% faster).</li></ul><h2>Real Business Benefits</h2><ul><li><strong>Faster responses</strong> – for agents processing tasks (e.g., Codex bots traversing workflows), we’re talking about saving tens of seconds per iteration, which daily adds up to <strong>4–6 hours saved</strong> for a team working in “agent loop” mode (e.g., 6 h × 5 people × 60 zł/h = <strong>1,800 zł daily</strong>).</li><li><strong>Reduced API costs</strong> – fewer requests mean lower token usage and smaller bills, especially in heavy document automation, mail processing, or CRM integrations.</li><li><strong>Better user experience</strong> – AI agents respond instantly, which builds trust and facilitates solution adoption within the company.</li></ul><h2>Technical Challenges – What Needs to Be Done to Make It Work?</h2><ul><li>Thoughtful connection-scoped caching design – in the implementations I’ve run, monitoring and optimizing memory for long sessions proved crucial.</li><li>Securing WebSocket connections – it’s important to pay attention to authorization and session handling to avoid context leaks.</li><li>Adapting processes and monitoring API handling – switching to WebSockets requires a different backend architecture.</li></ul><p>From my perspective: if you’re using AI agents in your company – it’s worth checking out the Responses API with WebSockets right now. It’s a concrete competitive edge at the production stage.</p><p><strong>Let me know in the comments if you’re implementing agent workflows with OpenAI, or write if you’d like to analyze savings for your company.</strong></p>
